Т.к. в условиях задания не дана информация об особенностях инфраструктуры и бюджета, на основе которой можно точнее выбрать программный продукт, то в качесте ответа предлагается использовать Kong. Использование плагинов позволяет подстроить его под разную инфраструктуру, делая Kong наиболее универсальным программным продуктом, который соответствует всем указанным в задании требованиям. На втором месте стоит упомянуть Nginx из-за простоты его развертывания и настройки, но поддержка только auth_basic при проверке аутентификции в бесплатной версии ставит его позади Kong, т.к. в задании не указано, какой должен использооваться механизм аутентификации, и, соответственно, Kong в этом плане выигрывает из-за универсальности плагинов.

## Задание 2

| Программный продукт | Производительность | Форматы сообщений | Разделение прав доступа | Кластеризация | Хранение сообщений на диске | Сложность эксплуатации |
| ------------------- | ------------------ | ----------------- | ----------------------- | ------------- | --------------------------- | --------------------- |
| Apache Kafka | Отличная (до 605 МБ/с пропускной способности) | Бинарные, JSON, Avro, Protobuf, любые сериализованные данные через схемы | Да (ACL для топиков, кластеров, групп; SASL/Kerberos аутентификация) | Да (реплицированные кластеры с ZooKeeper/KRaft для высокой доступности и отказоустойчивости) | Да (хранение на основе файловых логов с настраиваемым сроком хранения) | Средняя (сложная настройка требует экспертизы) |
| RabbitMQ | Хорошая (до 60k сообщений/с, но ниже 38 МБ/с с зеркалированием; низкая задержка для реального времени) | Бинарные, текст, JSON, XML, любые сериализованные объекты | Да (ACL для exchanges, очередей, vhosts; подключаемая аутентификация) | Да (кластеризация с зеркальными очередями, но чувствительна к сетевым разделениям) | Да (настраиваемое хранение на диске/ОЗУ для durable очередей) | Низкая (простая настройка с богатым UI/дашбордом, CLI инструментами) |
| Apache Pulsar | Высокая (до 305 МБ/с пропускной способности, 25 мс задержка) | Бинарные, JSON, Avro, Protobuf, любые через схемы | Да (JWT/Kerberos аутентификация) | Да (кластеризованные брокеры с geo-репликацией) | Да (хранение через BookKeeper) | Средняя (удобные CLI и дашборды для управления, требует запуска нескольких узлов) |

На основе собранных данных, в качестве ответа предлагается использовать Apache Kafka. Несмотря на то, что RabbitMQ проще в настройке, Kafka предлагает более высокую пропускную способность, особенно если учитывать потребность в кластеризации, где у RabbitMQ возникают проблемы с пропускной способностью.
